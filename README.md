개발 할 때 참고 해야 하는 사항 : 

  1. 건축으로 치면 개집을 지을 때는 설계도 없이 톱질부터 해도 되지만, 사람 사는 집을 지을 때는 최소한 기둥 위치(DB 설계)는 잡아야 무너지지 않습니다. (개집 만들고 싶지 않으면 잘 하라는 뜻)
  2. 구현 하려는 기능의 페이지를 간단하게 스케치라도 작성한다.
  3. 스케치 -> DB 설계 -> 구현 -> 테스트 코드 작성 및 테스트 -> API 문서 작성 -> 확인 (BEST)
  4. 테스트 코드 작성 안하고 체크 안하면 나중에 DB 뜯고 백엔드 뜯고 해야해서 일 두번 세번 해야하니까 그냥 편하게 만들고 테스트 하자.
  5. 하기 싫으면 좀 자거나 멍때리고 한다. 억지로 하면 더 하기 싫어짐 ( 폰 X , 뇌 과학적으로 폰 하면 뇌 써야해서 더 피곤하다고 함 그냥 멍때리거나 자는 게 좋다고 함)
  6. 거창하게 만들려고 하지 말고 하나 만들고 반응 보고 덧붙혀 가는 형식으로 구현한다. (실제로 크게 하려고 하면 머리 뜨거워져서 아예 내팽개침 죽어라 안 봄)
  7. 기능 개발 할 때 왜 이 기능이 필요한지 주변의 반응 보고 한다. (실제로 학원에서 센서들의 모든 값들을 각 디바이스마다 넣어서 받으려고 하였으나 주변 조언을 들어보니 그냥 메인으로 필요한 곳 한 곳만 박아두고 쓰고 위험관리 부분만 체크를 하면 유지보수성이나 여러 부분에서 좋다는 말을 듣고 깨달음)
  8. 왜 이걸 쓰는지 간단한 이유라도 꼭 생각하고 넘어가야함 아무 생각 없이 하면 나중에 진짜 이 기능을 사용해야 할 때 뭐 써야할 지 감도 안 잡 힘
  9. 사소한 디테일 좀 신경 쓰자 제발
  10. 한번 할 때 제대로 잡아두고 미루지 말자. 나중에 2~3번 보면 그게 더 시간이 오래 걸림

바이브 코딩 (Gemini , claude , perplexity , Antigravity) 시작.
  저번주에 claude + gemini + perplexity 를 활용하여 프로그램을 여태까지 진행해온 프로그램을 만들어 봤음. 프로젝트를 진행하는 구조는 perplexity (아키텍쳐 설계 및 프로그램 규칙 정의) -> gemini (실제 코드 구현) -> claude (코드 리팩토링) 
  각자 3개의 AI는 각각의 산출물들을 통하여 서로 데이터를 주고 받았음 (도메인들마다 일정한 품질을 유지하기 위함)
    제미나이 -> 클로드 AI로 가는 방향에서 클로드 AI 에게 리팩토링을 진행을 시켰으나 perplexity, gemini 에서 어느정도 확정 한 UI/UX 부분을 나의 의도에 맞지 않게  설계를 변경을 하였음. -> Gemini 부분에서 좋은 UI/UX 와 어플리케이션 테스트 부분에 테스트가 일정 이상 통과가 됐을 경우 클로드에   게는 UI/UX 부분을 아예 변경하지 않도록 강력하게 요청을 해야함.
    ai에게 범위가 넓은 부분의 기능을 요청을 하는 경우 그 요청을 한 유저의 기능과는 전혀 다른 결과물이 나올 확률이 높음. 요청과 상당히 유사한 기능을 받기 위해서는 상당히 자세한 정보 아키텍쳐 설계 , 간단한 목업 디자인 , 이런 부분의 내용을 어느정도 기술을 해주어야 좋은 결과가 나옴.
  처음은 react만 활용하여 간단한 기능을 맡겼고 요청의 범위도 엄청 작게 진행 -> Good ! 하지만 여기에서 느낀 것이 react만 활용하였을 경우 추후 다른 application 에 같은 기능을 넣어야 하는 경우 다시 또 설계를 해야하는 귀찮은 부분이 생기겠구나 라는 생각이 나옴.
  그렇기에 SpringBoot를 활용하여 API를 만들어서 react에 데이터를 뿌려주는 방식이 추후에 서비스 확장과 유지보수에 있어서 확실하게 좋은 부분이 생길 것 같다라고 생각됨. 또한 Spring 에서는 SpringAI를 지원을 해주기 때문에 AI를 활용한 새로운 도메인들이 생기기 떄문에 더욱 퀄리티가 좋은 어플리케   이션이 나올 확률이 높아짐.
